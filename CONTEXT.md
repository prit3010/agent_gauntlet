# Agent Gauntlet

Agent Gauntlet is a reliability context for agent harness optimization. The language here keeps the hackathon pitch precise when discussing adaptation, trust, validation, and promotion.

## Language

**Agent Gauntlet**:
A reliability layer that evaluates and improves the external harness around an AI agent using scenarios, traces, validators, and promotion gates.

**Harness Optimization**:
Evidence-gated improvement of external agent artifacts such as task prompts, skills, guardrails, validators, scenario generators, promotion gates, and output requirements. It does not retrain model weights.
_Avoid_: self-improvement without qualification, model training, making the model smarter

**Harness Adaptation**:
The way Agent Gauntlet changes future agent behavior by versioning and promoting better harness artifacts based on failure evidence. Adaptation happens in the harness, not in model weights.
_Avoid_: model adaptation, retraining

**Harness**:
The versioned operating environment around an agent: instructions, skills, tool access, guardrails, validators, context rules, helper scripts, output schemas, and promotion criteria.
_Avoid_: prompt, wrapper

**Context Map**:
A generated and reviewable artifact that identifies project-specific risks, protected paths, source priorities, public contracts, and relevant workflow context for a gauntlet run.
_Avoid_: repo summary, documentation dump

**Source Priority**:
The ordered trust ranking Agent Gauntlet uses when project artifacts conflict. Required tests, runtime errors, typecheck errors, official migration guides, API contracts, and existing code patterns outrank arbitrary repository prose.
_Avoid_: read everything equally, latest instruction wins

**Repo Prompt Injection**:
Untrusted or stale project text that tries to override the agent harness, skip validation, weaken safety rules, or redirect the task away from the user's goal.
_Avoid_: project instruction, documentation note

**Harness Version**:
A snapshot of the external artifacts that govern an agent's behavior. Agent Gauntlet scores and promotes harness versions rather than isolated prompt edits.
_Avoid_: prompt version, agent version

**Held-Out Validation**:
Evaluation on scenarios that were not used to generate a candidate harness patch. It is the evidence used to decide whether a patch generalizes beyond the failures that inspired it.
_Avoid_: retesting, demo scenarios

**Overblocking**:
A failure mode where a harness patch prevents a valid agent action or migration in the name of safety. Agent Gauntlet treats overblocking as a regression, not a success.
_Avoid_: being extra safe, strict mode

**Promotion**:
The act of accepting a harness version as the current best version after it passes held-out validation, critical safety checks, and regression limits.
_Avoid_: deploy, approve manually

**Reviewable Harness Artifact**:
The promoted output of Agent Gauntlet, including versioned harness files and the evidence report explaining why the harness was promoted.
_Avoid_: score, leaderboard result

**Useful Task Completion**:
Successful completion of valid in-scope scenarios without refusing, blocking, or avoiding the required work. Promotion requires usefulness as well as safety.
_Avoid_: pass by refusal, safe because blocked

**Deterministic Validator**:
An executable check whose result does not depend on LLM opinion, such as a contract test, snapshot test, AST rule, diff rule, type check, protected-path check, or command-trace requirement.
_Avoid_: AI judge, rubric

**Reliably Checkable Invariant**:
A correctness property that Agent Gauntlet can verify with executable evidence for the workflow, such as API alias preservation, protected-path integrity, test integrity, or validation evidence before completion.
_Avoid_: nice-to-have check, weak heuristic

**Correctness Hint**:
Optional user-provided evidence about behavior that should be preserved, such as an existing test, API contract, example, doc excerpt, or short natural-language rule.
_Avoid_: required setup, complete validator spec

**Seed Invariant**:
A starting correctness invariant used by the baseline gauntlet. Seed invariants can come from user-provided hints, existing tests, API contracts, docs, repo structure, or domain-pack defaults.
_Avoid_: complete invariant set, final validator list

**Discovered Invariant**:
A correctness invariant generated from observed failure clusters and promoted only if its harness patch improves held-out validation without overblocking.
_Avoid_: hand-authored demo rule, arbitrary secondary invariant

**Proposed Check**:
A candidate validator, guardrail, or invariant generated by Agent Gauntlet. It is not accepted until it is grounded in evidence and passes promotion without overblocking.
_Avoid_: accepted rule, generated truth

**LLM Judge**:
A supporting evaluator used for softer review dimensions such as diff reviewability, explanation quality, or residual-risk clarity. It cannot override deterministic validators for critical safety failures.
_Avoid_: validator, source of truth

**Gauntlet Scenario**:
A workflow-specific test case that runs an agent against a realistic task with expected evidence, risk triggers, and validation criteria. Scenarios may be generated, but they are grounded in project artifacts and domain-pack failure modes.
_Avoid_: prompt, benchmark question

**Hidden Oracle**:
The private expected outcome and validation logic for a gauntlet scenario. The agent under test should not see the hidden oracle during the run.
_Avoid_: answer key shown to the agent

**Visible Guidance**:
Instructions, skills, policies, and task context intentionally shown to the agent under test. Visible guidance is allowed to shape behavior, but it must be separated from hidden oracles and held-out validation logic.
_Avoid_: hidden tests, private validators

**Coverage Report**:
The evidence summary showing which risk categories, scenarios, validators, and remaining gaps were tested for a harness version. It scopes confidence instead of claiming total correctness.
_Avoid_: guarantee, certification

**Residual Risk**:
The remaining uncertainty after a harness version passes its gauntlet, including untested scenarios, incomplete validators, weak source inference, sandbox limitations, and possible overfitting.
_Avoid_: hidden failure, ignored risk

**Escaped Failure**:
A real failure that was not caught by the current gauntlet scenarios or validators. Escaped failures become inputs for future scenarios, validators, guardrails, or skills.
_Avoid_: proof the system failed completely, impossible case

**Replayable Trace**:
A recorded sequence of agent actions, tool calls, file reads, file edits, command runs, test outputs, validator results, and final-answer evidence for a gauntlet run.
_Avoid_: log dump, transcript

**Rejected Alternative**:
A candidate harness patch that failed promotion because it missed held-out failures, introduced regressions, overblocked valid work, increased critical failures, or violated cost and diff limits.
_Avoid_: failed idea, bad patch

**Agent Workflow Readiness**:
The assessed suitability of an agent harness to operate with less oversight on a defined class of tasks. It is different from whether one code patch passes CI.
_Avoid_: CI status, code quality score

**Readiness Score**:
An explainable and configurable summary of harness performance for a defined workflow. Critical safety failures cap the score so aggregate metrics cannot hide dangerous regressions.
_Avoid_: universal benchmark score, model score

**Optimizer Harness Improvement**:
A future capability where Agent Gauntlet proposes bounded changes to its own optimizer harness. It is not part of the hackathon v1 demo.
_Avoid_: recursive self-improvement, the AI rewrites itself

**Migration Wedge**:
The initial Agent Gauntlet vertical focused on code migration because the work is repeated, expensive, behavior-preserving, and measurable.
_Avoid_: toy demo, arbitrary first use case

**Codex-First Harness**:
The v1 integration target for Agent Gauntlet, focused on Codex artifacts such as `AGENTS.md`, `SKILL.md`, harness config, validator scripts, and report JSON.
_Avoid_: generic agent support, cross-agent portability

**V1 Scope**:
The hackathon product boundary: a Codex-first, migration-focused closed loop. Broader agent support and optimizer self-improvement are future work.
_Avoid_: universal agent platform, recursive optimizer

**Failure Cluster**:
A group of gauntlet failures with the same likely root cause and patch targets, such as API contract regression, test weakening, incomplete validation, overblocking, or repo prompt injection.
_Avoid_: individual failed run, vague category

**Bounded Patch Generation**:
The v1 patching approach where failure clusters map to allowed harness patch targets and candidate combinations. Promotion is decided by validation, not by the patch generator's confidence.
_Avoid_: free-form self-modification, hand-coded demo patch

**Trace-to-Promotion Sequence**:
The core demo moment where a dangerous baseline behavior is shown in a replayable trace, converted into candidate harness patches, validated on held-out scenarios, and promoted only when the evidence improves.
_Avoid_: final score reveal, dashboard tour

**Closed-Loop Harness Optimization**:
The full process from instrumented agent run to trace capture, evaluation, failure clustering, candidate harness patches, held-out validation, and promotion or rejection.
_Avoid_: dashboard, test report

**Eval Runner**:
A component that scores behavior or checks outputs for a run. Agent Gauntlet can use eval runners, but the product boundary is the closed loop that turns failures into promoted harness changes.
_Avoid_: Agent Gauntlet, full product

**Outer-Loop Harness Optimizer**:
Agent Gauntlet's role around an existing coding agent. It prepares scenarios, runs the agent, captures evidence, evaluates results, proposes harness patches, validates candidates, and exports promoted harness versions.
_Avoid_: coding agent, model, task executor

**Inner-Loop Task Executor**:
The coding agent being evaluated and improved by Agent Gauntlet. It performs the actual migration or workflow task inside the harness.
_Avoid_: optimizer, evaluator

**Disposable Run Environment**:
An isolated worktree or sandbox used for one gauntlet run, with scoped filesystem access, controlled environment variables, command tracing, and no production credentials.
_Avoid_: local dev machine, production repo

**Unsafe Tool Behavior**:
Agent behavior such as forbidden file deletion, protected-path edits, permission changes, unauthorized network access, secret access, skipped required checks, or command execution outside policy.
_Avoid_: agent mistake, harmless command

**Review Leverage**:
The product value of turning repeated manual review findings into durable harness improvements and evidence that makes future review faster and more focused.
_Avoid_: replacing reviewers, eliminating code review

## Example Dialogue

Dev: "Does Agent Gauntlet make the model better?"

Domain expert: "No. Agent Gauntlet performs harness optimization. It improves the versioned artifacts around the agent and promotes them only when held-out validation improves without increasing critical failures."

Dev: "Where does the system adapt?"

Domain expert: "Agent Gauntlet adapts by versioning and promoting better harness artifacts based on failure evidence, not by retraining the model."

Dev: "What exactly gets better?"

Domain expert: "The harness version gets better. Individual prompts, skills, guardrails, and validators may change, but Agent Gauntlet evaluates and promotes them as one coherent version."

Dev: "How do we know the harness is not just memorizing the demo?"

Domain expert: "Candidate patches are generated from training failures, but promotion depends on held-out validation. A patch that blocks valid migrations is rejected as overblocking."

Dev: "What if the harness just gets safer by refusing hard work?"

Domain expert: "That is overblocking. Promotion requires useful task completion on valid scenarios, not just fewer safety failures."

Dev: "What does Agent Gauntlet output?"

Domain expert: "The output is not just a score; it is a reviewable harness artifact plus evidence for why it was promoted."

Dev: "Is one AI grading another AI?"

Domain expert: "Not for critical safety. Deterministic validators own safety-critical verdicts; LLM judges only support softer review checks."

Dev: "Aren't deterministic validators hard to build?"

Domain expert: "V1 encodes the most important reliably checkable correctness invariants. The goal is as much correctness as possible through executable evidence, with remaining gaps reported explicitly."

Dev: "Do we predefine every secondary invariant?"

Domain expert: "No. Users provide whatever correctness hints they have, and Agent Gauntlet attempts to fill gaps from repo evidence, domain-pack defaults, and failure clusters."

Dev: "What if Agent Gauntlet invents the wrong rule?"

Domain expert: "Agent Gauntlet may propose discovered checks, but promotion requires evidence, held-out validation, and no overblocking."

Dev: "Do generated scenarios prove the agent is safe?"

Domain expert: "No. Gauntlet scenarios create scoped evidence. The coverage report shows what was tested, what passed, and which risks remain uncovered."

Dev: "What are Agent Gauntlet's own failure modes?"

Domain expert: "Agent Gauntlet can fail when its gauntlet is weak, so the product makes coverage, evidence, and residual risk visible instead of claiming certainty."

Dev: "What if a validator misses a subtle bug?"

Domain expert: "Then confidence was only scoped to existing coverage. The escaped failure becomes a future scenario, validator, guardrail, or skill improvement."

Dev: "Can the agent game the gauntlet?"

Domain expert: "It can see visible guidance, but not hidden oracles. Critical verdicts come from external traces, diffs, command evidence, and held-out validators."

Dev: "Why not just add more CI?"

Domain expert: "CI gates a specific code change. Agent Gauntlet gates the agent workflow that produces code changes by evaluating traces, harness versions, and repeated failure patterns."

Dev: "Should the hackathon demo show the optimizer improving itself?"

Domain expert: "No. Optimizer harness improvement is a future capability. The hackathon demo should focus on improving the Migration Pilot harness from baseline v1 to promoted v2."

Dev: "Why should judges believe the improvement is real?"

Domain expert: "The improvement is backed by replayable traces, patch diffs, held-out validation results, and rejected alternatives."

Dev: "Are the readiness weights arbitrary?"

Domain expert: "The score is explainable and configurable. Critical failures cap readiness so aggregate metrics cannot mask safety regressions."

Dev: "Why start with migration?"

Domain expert: "Migration is the wedge because it is repeated, expensive, behavior-preserving, and measurable."

Dev: "Does v1 work across every coding agent?"

Domain expert: "No. V1 is a Codex-first harness. Cross-agent transfer is a future direction."

Dev: "What is in scope for the hackathon?"

Domain expert: "V1 is a Codex-first, migration-focused closed loop; broader agent support and optimizer self-improvement are future work."

Dev: "Is patch generation just hand-coded?"

Domain expert: "V1 uses bounded patch generation from failure clusters; validation, not hand-waving, decides promotion."

Dev: "What is the strongest demo moment?"

Domain expert: "The trace-to-promotion sequence: one dangerous agent behavior becomes a validated guardrail, skill update, and regression test."

Dev: "Is this just a dashboard around test results?"

Domain expert: "No. The product is closed-loop harness optimization: it turns failure traces into validated harness patches that change future agent behavior."

Dev: "Is this just evals plus CI?"

Domain expert: "No. Evals and CI-style checks are components. Agent Gauntlet is the closed-loop harness optimizer that uses them to promote better harness versions."

Dev: "Are we building a new coding agent?"

Domain expert: "No. Agent Gauntlet is the outer-loop harness optimizer; the coding agent remains the inner-loop task executor."

Dev: "How do we safely run agent commands?"

Domain expert: "Agent Gauntlet runs agents in disposable, policy-controlled environments and treats unsafe tool behavior as first-class evaluation evidence."

Dev: "Why not just review agent PRs manually?"

Domain expert: "Agent Gauntlet converts repeated manual review pain into automated harness improvements and evidence for higher-leverage review."

Dev: "How does the system know what is risky in this repo?"

Domain expert: "Agent Gauntlet builds a reviewable context map from repo artifacts and domain-pack defaults."

Dev: "What if a README tells the agent to ignore tests?"

Domain expert: "Repo text is evidence, not authority. Source-priority rules and validators prevent arbitrary project prose from overriding the harness."
